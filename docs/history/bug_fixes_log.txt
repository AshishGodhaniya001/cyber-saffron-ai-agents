Fix #1: REMOVED RECURSIVELY NESTED app/ DIRECTORY COPIES
------------------------------------------------------------
Location: app/app/ (and all deeper nested copies)
Severity: CRITICAL
Description:
  The project had accidental recursive copies of the entire app/
  directory nested inside itself (up to 6+ levels deep). This
  caused:
    - Python import resolution conflicts (from app.config import
      settings could resolve to the nested copy)
    - Bloated project size (hundreds of duplicate files)
    - Unpredictable behavior depending on Python's sys.path
    - Difficulty maintaining the codebase
Fix:
  Removed the entire app/app/ directory tree, keeping only the
  top-level app/ files as the single source of truth.

------------------------------------------------------------
Fix #2: MASTER CTO AGENT - SESSION SCOPE BUG
------------------------------------------------------------
Location: app/agents/master_cto.py (lines 30-68)
Severity: CRITICAL
Description:
  _orchestrate() created a DB session via 'async with' and passed
  it to background tasks via asyncio.create_task(). The session was
  automatically closed when _orchestrate() finished, but the
  background tasks (_execute_task) ran after the session was
  already closed. This caused:
    - SQLAlchemy errors ("Background operation on closed session")
    - Tasks stuck in "running" state forever after first error
    - No error propagation to the database
Fix:
  Each background task now opens its own independent session via
  'async with AsyncSessionLocal() as db:', ensuring the session
  lives as long as the task runs. Also added proper exception
  handling with two try/except blocks and a rollback-safe flow.

------------------------------------------------------------
Fix #3: LEAD GEN AGENT - DOUBLE SESSION OVERHEAD
------------------------------------------------------------
Location: app/agents/lead_gen.py (lines 23-51)
Severity: MEDIUM
Description:
  _discover_leads() opened two separate DB sessions sequentially
  when one sufficed. The first session saved leads and committed,
  then a second session created follow-up tasks. This was
  inefficient and risked partial failures.
Fix:
  Merged both operations into a single session. Leads and tasks
  are now created in the same transaction, and the commit happens
  only at the end.

------------------------------------------------------------
Fix #4: MISSING ENGINE DISPOSAL ON SHUTDOWN
------------------------------------------------------------
Location: app/main.py (line 15)
Severity: MEDIUM
Description:
  The FastAPI lifespan shutdown handler stopped the scheduler but
  never disposed the SQLAlchemy async engine. This caused:
    - Unclosed database connections on shutdown
    - Potential connection pool exhaustion during restarts
    - Resource leaks in Docker environments
Fix:
  Added 'await engine.dispose()' after stop_scheduler() in the
  lifespan shutdown handler.

Fix #5: EXPOSED API KEYS IN .env
------------------------------------------------------------
Location: .env
Severity: CRITICAL (security)
Description:
  The .env file contained real, live API keys:
    - GEMINI_API_KEY (Google Gemini API)
    - RESEND_API_KEY (Resend email service)

    - RESEND_API_KEY=your_resend_api_key_here
    - EMAIL_FROM=onboarding@resend.dev
  Also added .env to .gitignore so it won't be committed again.

------------------------------------------------------------
Location: app/models.py (line 37)
Severity: LOW (consistency)
Description:
